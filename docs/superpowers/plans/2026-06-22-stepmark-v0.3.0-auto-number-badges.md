# StepMark V0.3.0 Auto Number Badges Implementation Plan

> **For agentic workers:** Implement task-by-task in order. Each task should read the V0.3 spec plus only the task section being executed. Use checkbox syntax for tracking. Make one commit per completed task.

**Goal:** Add automatic global number badges for smart note, rectangle/ellipse, arrow, and text annotations. Smart note numbering is on by default; other tools are opt-in. Number settings persist, but each screenshot starts from 1.

**Design:** `docs/prds/stepmark-v0.3.0-design.md`

**Prerequisite:** V0.2.0/V0.2.2 editor and style panel work is complete.

## Task 1: Add numbering types and persistent settings store

**Goal:** Define badge data, placement settings, defaults, and localStorage-backed validation.

**Files:**
- Modify: `src/types/annotation.ts`
- Create: `src/types/numbering.ts`
- Create: `src/store/numberingStore.ts`
- Create: `src/store/numberingStore.test.ts`

- [ ] **Step 1: Extend annotation types**

Add `BadgeShape`, `NumberBadgeStyle`, `NumberBadge`, and optional `numberBadge?: NumberBadge` to `src/types/annotation.ts`.

Append `numberBadge` to the existing interface instead of reordering current fields.

- [ ] **Step 2: Add numbering setting types**

Create `src/types/numbering.ts` with:

- `NumberedTool`
- `RectBadgePosition`
- `EllipseBadgePosition`
- `ArrowBadgePosition`
- `TextBadgePosition`
- `SmartBadgeAnchor`
- `SmartBadgePlacement`
- `TargetBadgePlacement`
- `NumberingSettings`
- `DEFAULT_NUMBERING_SETTINGS`

`NumberedTool` must be a fixed 4-key union. Do not derive it from `ToolType`, because `ToolType` also contains `select` and `mosaic`, which must not participate in numbering.

Defaults must match the spec:

- smart enabled, rect/arrow/text disabled.
- smart placement = label-left.
- badge style = blue square, white text, 13px.

- [ ] **Step 3: Create `numberingStore`**

Use Zustand + localStorage:

```typescript
const STORAGE_KEY = "stepmark.numbering.v1";
```

Store API:

```typescript
interface NumberingState {
  settings: NumberingSettings;
  updateSettings: (patch: Partial<NumberingSettings>) => void;
  updateEnabled: (tool: NumberedTool, enabled: boolean) => void;
  updateToolPlacement: <K extends keyof NumberingSettings["positionByTool"]>(
    tool: K,
    placement: NumberingSettings["positionByTool"][K]
  ) => void;
  updateBadgeStyle: (patch: Partial<NumberBadgeStyle>) => void;
  resetSettings: () => void;
}
```

Keep the exact API pragmatic if the existing store style suggests a smaller shape, but preserve the same capabilities.

Do not merge numbering settings into `toolState.ts`. That store is ephemeral drawing state; numbering settings are persistent user preferences.

- [ ] **Step 4: Validate persisted settings**

Field-level fallback:

- colors: `#[0-9a-fA-F]{6}`
- font size: integer `8-72`
- shape: `square | rounded | circle`
- placement: valid union member
- enabled: boolean

Do not throw on corrupt localStorage.

- [ ] **Step 5: Unit tests**

Cover:

- no localStorage -> defaults
- valid persisted settings -> loaded
- corrupt JSON -> defaults
- partially invalid settings -> field-level fallback
- updates persist

- [ ] **Step 6: Verify**

```bash
npm test -- numberingStore
npm run build
```

- [ ] **Step 7: Commit**

```bash
git add src/types/annotation.ts src/types/numbering.ts src/store/numberingStore.ts src/store/numberingStore.test.ts
git commit -m "feat: add numbering settings store"
```

## Task 2: Add editor sequence state and numbering application helper

**Goal:** Make `nextNumber` deterministic across create, undo, redo, delete, and reset.

**Files:**
- Modify: `src/store/editorStore.ts`
- Create: `src/numbering/applyNumbering.ts`
- Create: `src/numbering/applyNumbering.test.ts`

- [ ] **Step 1: Add `nextNumber` to editor state**

In `src/store/editorStore.ts`:

- Add `nextNumber: number`.
- Set `nextNumber: 1` in initial state.
- Set `nextNumber: 1` in `init(...)`.
- Add `resetNextNumber: () => void`.

- [ ] **Step 2: Add `nextNumber` to history snapshots**

Extend `EditorSnapshot`:

```typescript
interface EditorSnapshot {
  annotations: Annotation[];
  cropRegion: Rect;
  nextNumber: number;
}
```

Update `snapshot(s)` and undo/redo restore paths.

The affected paths are `snapshot`, `addAnnotation`, `updateAnnotation`, `removeAnnotation`, `undo`, and `redo`. `updateAnnotation` and `removeAnnotation` should include `nextNumber` in their snapshots but must not change it directly.

- [ ] **Step 3: Support number consumption on add**

Update add flow so numbered annotations increment `nextNumber`, while unnumbered annotations do not.

Recommended approach:

```typescript
addAnnotation: (a: Annotation, options?: { consumedNumber?: boolean }) => void;
```

If `options?.consumedNumber === true`, set `nextNumber: s.nextNumber + 1`.

- [ ] **Step 4: Keep delete/update neutral**

Confirm:

- `removeAnnotation` does not change `nextNumber`.
- `updateAnnotation` does not change `nextNumber`.
- `resetNextNumber` does not change existing annotations.
- `resetNextNumber` does not push an undo/redo history entry.

- [ ] **Step 5: Add numbering helper**

Create `src/numbering/applyNumbering.ts`:

```typescript
export function applyNumberBadgeIfEnabled(...)
```

It should:

- return original annotation and `consumed=false` when disabled.
- add `numberBadge.value = nextNumber` and a copied style when enabled.
- not mutate inputs.

- [ ] **Step 6: Unit tests**

Cover:

- disabled tool does not add badge or consume.
- enabled tool adds expected value/style and consumes.
- style is copied, not referenced.
- editor undo/redo restores `nextNumber` if editor store tests are already established; otherwise add focused tests where practical.

- [ ] **Step 7: Verify**

```bash

npm run build
```

- [ ] **Step 8: Commit**

```bash

git commit -m "feat: track numbering sequence"
```

## Implementation Notes

- Keep pure geometry in `src/geometry/numberBadge.ts`.
- Keep persisted settings in `src/store/numberingStore.ts`.
- Keep session sequence in `src/store/editorStore.ts`.
- Badge value/style are stored on the annotation; badge placement is read from current settings during render.
- Do not let disabled tools consume numbers.
- Do not add badge logic to mosaic.
- Do not render badges as HTML overlays.
- Do not make badges independently draggable in V0.3.0.
