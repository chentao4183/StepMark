## 0. 总体结论

**Overall: WARN（可发布，但有若干需关注的偏离点）**

V0.2.0 PRD 的核心目标基本达成：去阴影、crop 可调、智能标注几何重构、每工具样式持久化、二级样式面板全部落地，34 个单测通过、`npm run build` 通过。

最值得关注的是 **坐标模型的实现方式偏离了 design.md 的字面规定**（但 Codex 在 design.md §2.1 自己留了一个"实现说明"口子让这种偏离合法化）—— 实际实现采用的是"全屏原图 + Konva stage 直接等于 source space + clip 裁剪"的简化模型，没有引入 `sourceToViewport` / `viewportToSource` 转换层。这个偏离**对功能无影响**（当前截图是 1:1 像素贴在窗口上），但会让后续 V0.2.1 spec 想要的"crop 可平移、背景随 crop 滑动"无法直接复用。

其余偏离点都是默认值/数值范围的小差异，以及一个 mosaic 选中色未统一的小 bug。

## 4. 主要偏离点详述

### 4.1 ❌ 坐标转换层缺失（最重要的发现）

**PRD/design 规定**（design.md §2.3 + §10.1）：

> All tool hooks and shape renderers must use explicit conversion helpers instead of local ad hoc math.
> ```ts
> function sourceToViewport(point, crop) { return { x: point.x - crop.x, y: point.y - crop.y }; }
> function viewportToSource(point, crop) { return { x: point.x + crop.x, y: point.y + crop.y }; }
> ```
> Required unit tests: `sourceToViewport`, `viewportToSource`.

**实际实现**：完全没有这两个函数。整个编辑器用的是另一种模型：

- Konva Stage 大小 = `window.innerWidth/Height`
- 背景图以 `(0,0)` 全屏绘制，**不平移**（`EditorStage.tsx:49`）
- 标注坐标 = source 坐标 = stage 坐标（三者重合）
- crop 仅作为 clip 区域 + 导出区域（`AnnotationLayer.tsx:20`、`exportCanvas.ts:28`）
- 工具 hook 拿到 pointer 后直接当 source 坐标存（`useRectTool.ts:15 pos()` 直接返回 `getPointerPosition()`）

**Codex 在 design.md §2.1 自己留了个口子**：

> Implementation note for the current V0.1 codebase: the editor already draws the full screenshot at window size and treats Konva stage coordinates as screen/source coordinates. V0.2.0 should preserve that model and upgrade the existing initial selection rectangle into a mutable `cropRegion`; **it should not rewrite the editor into a separate crop-origin viewport coordinate system.**

所以**严格来说不违规** —— Codex 选了 design.md 自己提供的"B 方案"。但要注意：

1. 这跟 design.md §2.3（同文档下一节）的"explicit conversion helpers"自相矛盾 —— §2.1 说不要转换，§2.3 又说必须用转换 helper。实现跟了 §2.1。
2. PRD §Quality Standards 明确列了 `sourceToViewport` / `viewportToSource` 单测为 required，这部分**确实没满足**。
3. **与 V0.2.1 superpowers spec 冲突**：V0.2.1 spec §2.4-2.6 要求背景以 `(-cropRegion.x, -cropRegion.y)` 放置、stage 原点 = crop 左上角、crop 可平移（§7.5）。当前实现背景固定在 (0,0)，**无法支持 crop 平移**（crop 只能 resize 不能 move）。这是后续 V0.2.1 落地时需要二次重构的点。

**影响评估**：
- 对 V0.2.0 功能：无影响。当前 crop 只 resize、不平移，背景不动也能正确工作。
- 对 V0.2.1：需要补坐标转换层。不算欠债，因为 V0.2.1 spec 本来就要重构这块。
- 对代码质量：少了一层抽象，**当前反而更简单**（没有冗余的恒等转换函数）。属于合理的简化。

**结论**：记为 FAIL（因为 PRD Quality Standards 明确要求且没做），但严重程度低 —— 建议在 V0.2.1 重构时一并补上，或者现在补一个 `coordinates.ts`（即使只是恒等函数）来满足文档一致性。

### 4.2 ⚠️ 默认值偏离

| 参数 | PRD/design.md 规定 | 实际代码 | 偏离程度 |
|------|------------------|---------|---------|
| `arrowHeadSize` 默认 | **12**（design.md §7.2） | **10**（`toolStyle.ts:54`） | ⚠️ 小 |
| `strokeWidth` 范围 | 1-12（design.md §7.3 "recommended 1-12"） | 1-8（`toolStyle.ts:64`） | ⚠️ 小（design 说"recommended"） |
| `fontSize` 范围 | 10-72（design.md §7.3） | 8-72（`toolStyle.ts:65`） | ⚠️ 小 |
| `arrowHeadSize` 范围 | 6-40（design.md §7.3） | 4-20（`toolStyle.ts:66`） | ⚠️ 中 |
| `fontFamily` 默认 | `"system-ui"`（design.md §7.2） | `""`（`toolStyle.ts:43,59`） | ✅ 故意 —— 空=跟随系统，与 V0.2.2 superpowers spec §2.2 一致，是更好的设计 |

**注意**：V0.2.2 superpowers spec（更晚、更权威）写的 `arrowHeadSize` 默认就是 **10**、范围 **4-20**，跟实际代码一致。所以代码跟的是 V0.2.2 spec 而非 PRD。**建议以 V0.2.2 spec 为准**（AGENTS.md 说"分歧以 superpowers/specs 为准"），这部分不算 bug，但 design.md §7.2/7.3 的数值已过时，建议同步更新 design.md 避免后人混淆。

### 4.3 ⚠️ 几何 helper 命名不符

| PRD/design 接口名 | 实际函数名 | 文件 |
|------------------|-----------|------|
| `nearestPointOnRectBoundary` | `nearestRectEdgePoint` | `rectEdge.ts` |
| `pointOnEllipseBoundary` | `ellipseBoundaryPoint` | `ellipse.ts` |

功能等价，只是命名。design.md 是"预期 helper 名"，代码用了更短的名字。**不影响功能**，但如果有人按 design.md 找函数会找不到。建议二选一统一（改 design.md 或加 alias）。

### 4.4 ⚠️ MosaicShape 选中色未统一

`MosaicShape.tsx:85` 选中虚线框用的是 **`#00d2ff`**（浅亮蓝），而其他三个 shape（RectShape:25、ArrowShape:68、SmartAnnotationGroup:71、TextLabelShape:90）统一用的是 **`#1e90ff`**。

这违反 design.md §6.1 "selection feedback ... must be clearly an editing affordance" 的一致性要求 —— 同一编辑器里选中色有两种蓝，视觉不统一。而且 `#00d2ff` 正是 V0.1 阴影色（`shadowColor="#00d2ff"`）同色系，容易让人误以为阴影没去干净。

**这是 MosaicShape 在 `0e01936` 去 shadow 时漏改的**（diff 显示那次只动了 RectShape/ArrowShape/TextLabelShape/SmartAnnotationGroup，没动 MosaicShape）。建议改为 `#1e90ff` 与其他 shape 一致。

### 4.5 ⚠️ 智能标注箭头预览的 headSize 硬编码

`EditorStage.tsx:73-74` 智能标注**预览**箭头的 `pointerLength/pointerWidth` 硬编码为 `10`，没有读 `style.arrowHeadSize`（smart 工具的样式里本来也没有 arrowHeadSize 字段，这是合理的）。但与实际提交后渲染的 `a.arrowHeadSize ?? 10`（SmartAnnotationGroup:25）数值巧合一致（都是 10），**当前没问题**。

只是要注意：smart 工具的箭头头大小**没有进入样式系统**（V0.2.2 spec 也没把 smart 的 arrowHeadSize 列入面板），用户无法调智能标注的箭头头大小。这是 spec 决策（smart 的箭头是附属元素），不是 bug，但值得记录。

## 6. 风险提示（给后续 V0.2.1）

基于本次 review，V0.2.1 落地时需要特别注意：

1. **坐标转换层必须重建**（见 §4.1）：当前背景固定 (0,0) 模型无法支持 V0.2.1 spec §2.5 的"背景以 `(-crop.x,-crop.y)` 放置"和 §7.5 的 crop 平移。V0.2.1 的坐标重构会动到 EditorStage、所有 shape、所有 tool hook 的 pointer 处理 —— 是 V0.2.1 最高风险项。
2. **`BoundableShape` 没拆出来**：design.md §8.2 和 V0.2.1 §8.2 都建议把 RectShape 拆成 `BoundableShape`（rect/ellipse 二合一），实际代码是在 `RectShape.tsx` 内部用 if 分支处理的（`RectShape.tsx:17,32`），没单独建文件。功能等价，但 V0.2.1 如果要按 spec 拆，会有一轮文件移动。
3. **椭圆 smart 的箭头起点**：`arrowAnchor.ts smartArrowStart` 对 ellipse 调 `ellipseBoundaryPoint`，但**没有 V0.2.1 §5.1 要求的"外移 6px"**（OUTSET）。V0.2.1 需要补这个外移逻辑。
4. **smart 标注没有缩放能力**：当前选中只支持拖动移动，没有 8 拉手缩放（V0.2.1 A6 要求）。这是预期的分期，不是 bug。
