## 0. 总体结论

**Overall: PASS with caveats（文档方向正确，可作为 V0.3.0 实施依据，但有 8 个点需要在 plan 阶段澄清）**

V0.3 的两份文档方向一致、技术路径可行，PRD 自评 94/100 基本属实。核心结论：

1. **前置依赖已满足**：V0.3 文档声称复用的 V0.2 基础设施（全屏坐标模型、二级样式面板、4 个 shape 组件、toolStyleStore 模式）**全部存在于代码中**，无依赖阻塞。
2. **形态决策已确认**：用户在 brainstorming 中已确认采用「独立 badge」方案（序号作为飘在标注旁的独立小方块/圆，与标签文字分离），与 PRD 的核心定位一致。

4. **发现 4 个 design↔PRD 表述偏离 + 4 个文档↔代码不一致**，需要在 writing-plans 前明确处理方式（详见第 3、4 节）。

## 1. 文档方向性确认（与用户意图对齐）

用户最初描述的需求是「智能标注工具……自动序号标注……不需要手动输入序号」，**隐含的是「序号作为标签前缀」形态**。V0.3 文档走的是「独立 badge」形态（PRD L17：「Render the number as an independent badge attached to the annotation, **not as part of the annotation text**」）。

**两条形态路线对比**：

| 维度 | 用户最初设想（前缀式） | V0.3 文档方案（独立 badge） |
|------|----------------------|---------------------------|
| 序号位置 | 标签文字的一部分（如 `1. 按钮 disabled`） | 标注旁的独立小方块/圆 |
| 视觉风格 | 沿用红底标签 | 新增蓝色（`#1677ff`）badge |
| 工具覆盖 | 仅 smart | smart + rect + arrow + text |
| 工作量 | ~1 天（小功能） | 4-5 天（5 个 Phase） |

**决策**：用户在 brainstorming 中已选「独立 badge（V0.3 文档方案）」。因此本报告的核查基准为 PRD，核查重点是 design doc 是否偏离 PRD、以及两份文档是否会与现有代码冲突导致返工。

## 3. design doc ↔ PRD 不一致清单（4 处）

按严重程度排序。

### 🔴 3.1【严重，疑似 PRD 内部矛盾】`nextNumber` 持久化规则措辞

**位置**：PRD L42-43

**原文**：
> - Numbering settings persist across app restarts.
> - The current `nextNumber` value does not persist across screenshots or app restarts.

**问题**：两条 bullet 用句号连写，「settings persist」与「nextNumber does not persist」放在一起容易让实现者误读成"nextNumber 也属于 settings，需要持久化"。design doc §3.3 / §7.2 明确 `nextNumber` 只进 editorStore session、不持久化，与 PRD 后半句一致 —— **design doc 站得住，但 PRD 这条本身表述不清**。

**建议处理**：writing-plans 时以 design doc 为准（nextNumber 仅 session 级），并在 plan 中显式标注此口径。PRD 是否修订由用户决定。

### 🟡 3.2【表述偏离】Smart 工具默认锚点位置的措辞暗示错误形态

**位置**：design doc §4.6 最后一段

**原文**：
> Default smart note placement is: anchor: label, label position: left.
> This matches the intended "**number + explanation label**" workflow while still allowing users to attach badges to the target shape or arrow.

**问题**：「number + explanation label」的措辞**暗示序号和文字在同一标签里**（即前缀式形态），与 PRD「独立 badge」的核心定位（PRD L17）冲突。design doc §3.3 的默认值本身（`anchor: "label"`, `labelPosition: "left"`）没问题，只是这段解释文字误导。

**建议处理**：plan 实施时忽略这段解释，以 §3.3 类型定义为准；或在 plan 写作前请用户修订 design doc 该段。

### 🟡 3.3【决策越权】Reset 是否进 undo 历史

**位置**：design doc §3.4 末尾

**原文**：
> Recommendation: keep reset outside annotation undo history for V0.3.0. It affects only future creations and does not modify the canvas.

**问题**：PRD L60-62 / Acceptance 只规定 reset 的行为（不改现有 badge），**完全没提** undo。design doc 在此处主动做了一个 PRD 未授权的决策。这个决策本身合理（reset 不改画布，逻辑上不该进 undo），但属于 PRD 的遗漏。

**建议处理**：plan 中明确写"reset 不进 undo 历史"作为实施约束；若用户希望 PRD 也补这条，则同步修订 PRD Acceptance。

### 🟡 3.4【措辞模糊】"Square" 形状的渲染规则未在 PRD 定义

**位置**：PRD Acceptance L217 vs design doc §4.1

**对照**：
- PRD L217：「Badge shape supports square, rounded rectangle, and circle, with **square as the default**」—— 只说支持 square，没说 square 的几何含义
- design doc §4.1：「Square uses `size = max(width, height, BADGE_MIN_SIZE)` and renders **width = height = size**」—— 即 square 强制渲染为**正方形**（等宽等高）

**问题**：若实现者按 PRD 字面理解成"square = width 跟随文字、height 跟随字号"，会跟 design 不一致。design 的"正方形"语义更合理，但 PRD 没说。

**建议处理**：以 design doc §4.1 为准（square = 正方形），plan 中显式记录此几何规则。

## 4. 文档 ↔ 现有代码不一致清单（4 处，潜在返工点）

按严重程度排序。

### 🟠 4.1【潜在返工】`ToolType` 含 `"mosaic"`，但 PRD 明确排除 mosaic

**代码现状**（`src/types/annotation.ts:1`）：
```ts
export type ToolType = "select" | "smart" | "rect" | "arrow" | "text" | "mosaic";
```

**文档要求**：
- PRD L25：「Excluded: **mosaic** numbering」
- design doc §3.3 `enabledByTool` 只列 smart/rect/arrow/text，与 PRD 一致 ✅

**返工风险**：实现时若用 `Record<ToolType, ...>` 或遍历 `ToolType` 来定义 `enabledByTool`，会**多出 `select` 和 `mosaic` 两个不该有编号的 key**，需要写额外逻辑排除。design doc 没显式说"`enabledByTool` 是固定 4-key record，不要从 `ToolType` 派生"。

**建议处理**：plan 中明确 `enabledByTool` 的类型为固定 4 键 record，不使用 `Record<ToolType, boolean>`。

### 🟠 4.2【潜在返工】`EditorSnapshot` 加 `nextNumber` 需同步改 6 个函数

**代码现状**（`src/store/editorStore.ts:5-8, 33-130`）：

```ts
interface EditorSnapshot {
  annotations: Annotation[];
  cropRegion: Rect;
}
```

`snapshot()` 函数只拷贝 `annotations` + `cropRegion`，`undo()` / `redo()` 也只读这两项。

**文档要求**（design doc §3.4）：
```ts
interface EditorSnapshot {
  annotations: Annotation[];
  cropRegion: Rect;
  nextNumber: number;  // 新增
}
```

**返工风险**：design doc 提了要加 `nextNumber`，但**没给迁移说明**。下列 6 个函数都要改：
- `snapshot()` —— 加 `nextNumber` 字段
- `addAnnotation()` —— 入库前要 consume number
- `updateAnnotation()` / `removeAnnotation()` —— snapshot 带 `nextNumber`（这两个不改 nextNumber，但 snapshot 要带）
- `undo()` / `redo()` —— 读 `nextNumber` 恢复

另外 V0.2 写出的 snapshot（如果有持久化的话）没有 `nextNumber` 字段，需要兼容旧数据。

**建议处理**：plan 中把 `editorStore` 的改造拆成独立 Task，并明确"missing `nextNumber` 字段视为 `1`"的兼容策略。当前 `editorStore` 不持久化（init 每次重置），所以向后兼容问题可忽略，但要在 plan 里写清楚。

### 🟡 4.3【cosmetic】`Annotation` 类型字段顺序与代码不一致

**对照**：
- design doc §3.1 字段顺序：`id, type, rect, shape, note, arrow, style, lineStyle, arrowHeadSize, fontFamily, numberBadge`
- 代码 `src/types/annotation.ts:43-54` 实际顺序：`id, type, rect, shape, lineStyle, arrowHeadSize, fontFamily, note, arrow, style`

**返工风险**：字段内容完全一致，只是顺序不同。**纯 cosmetic，不会返工**，但如果 plan 让 subagent 严格对照 design 的字段顺序，会产生无意义的 diff。

**建议处理**：plan 中注明"以现有代码字段顺序为准，`numberBadge` 追加到末尾"。

### 🟡 4.4【待确认】`toolState.ts` 的职责与 `numberingStore.ts` 的关系

**代码现状**：`src/store/toolState.ts` 存在（Glob 到），但本次核查未读其内容。design doc §7.1 只提到要新建 `src/store/numberingStore.ts`，**没说要不要复用或合并 `toolState.ts`**。

**返工风险**：若 `toolState.ts` 已经管"每工具的持久化设置"（跟 numberingSettings 职责重叠），新建 `numberingStore.ts` 可能造成两个 store 职责混乱。若 `toolState.ts` 只管工具切换的 session 状态，则两者职责分离，新建 `numberingStore.ts` 合理。

**建议处理**：writing-plans 前先读 `toolState.ts` 内容，明确其职责后再决定是新建独立 store 还是合并。

## 6. 风险评估

**无阻塞性风险**。所有发现项都是表述/口径层面的问题，不影响 V0.3.0 整体可实施性。

**最大风险点**：第 4.1 和 4.2 项（mosaic 排除、editorStore 改造）若 plan 阶段不显式处理，subagent 执行时容易踩坑，导致：
- 4.1：编号意外应用到 mosaic/select，需要补丁式修复
- 4.2：undo/redo 序号状态错乱，需要回溯改 editorStore

**建议**：writing-plans 时把第 3 节（design↔PRD 口径）作为"实施约束"前置写入 plan 的 Note 块；把第 4 节（文档↔代码）作为具体 Task 的输入说明。
