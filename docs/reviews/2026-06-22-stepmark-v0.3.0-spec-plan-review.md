## 2. 建议修正项（3 处，描述与代码不符）

这些是新发现的、spec/plan 自身的问题，会让 subagent 执行时困惑。**建议在交付实施前修订 spec 或 plan**。

### 🔴 2.1【必修】spec §7.2 "坐标转换"描述与 V0.2 实际模型不符

**spec 原文**（§7.2）：
> - badge 使用 source space 计算，再跟随 V0.2 坐标转换进入 viewport/stage。

**代码实际**（`src/canvas/EditorStage.tsx:46-51`）：
```tsx
{/* Background is the full screenshot. Tools operate in screen/source
    coordinates, while cropRegion controls clipping, shade, and export. */}
<KonvaImage image={image} x={0} y={0} width={window.innerWidth} height={window.innerHeight} />
```

V0.2 的实际模型是**简化模型**（前置报告 `2026-06-18-stepmark-v0.2.0-codex-review.md` §0 已记录）：背景图直接 1:1 贴在 stage 原点，**stage space == source space == screen space**，**没有 `sourceToStage` / `stageToSource` 转换层**（V0.2.1 design §2.4 设想的转换函数未实现）。

**影响**：spec §7.2 这句话方向对（badge 用 source 坐标），但"跟随 V0.2 坐标转换"的措辞会让 subagent 去找不存在的转换函数，浪费排查时间。

**建议修正**：把 spec §7.2 第三条改为：
> - badge 直接使用 source space 坐标渲染。V0.2 当前 stage space 与 source space 重合（背景图 1:1 贴在 stage 原点），无需坐标转换。

### 🟠 2.2【必修】spec §6.1 measureText 回调来源未交代，且现有 measureTextWidth 未 export

**spec 原文**（§6.1）：
```typescript
export function measureNumberBadge(
  value: number,
  style: NumberBadgeStyle,
  measureText: (text: string, fontSize: number) => number
): BadgeBox
```

**代码实际**（`src/canvas/labelMetrics.ts:18`）：
```typescript
function measureTextWidth(text: string, fontSize: number, fontFamily: string | undefined): number {
  // 未 export，且签名带 fontFamily 参数
```

**问题**：
1. spec §6.1 把 `measureText` 设计成回调注入（为了可单测），这是好的。但 plan 和 spec 都没说**这个回调从哪儿来** —— 是 export 现有的 `measureTextWidth`？还是新建一个 badge 专用测量？还是 plan Task 3 现场决定？
2. 现有 `measureTextWidth` 签名带 `fontFamily` 参数，spec §6.1 的回调签名 `(text, fontSize) => number` **不带 fontFamily**。badge 的 `NumberBadgeStyle`（spec §3.1）也没有 `fontFamily` 字段。

**影响**：subagent 执行 Task 3 时会卡在"badge 文字用什么字体测量"这个问题上，可能随意决定，导致 plan Task 4 渲染时测量和实际渲染字体不一致（badge 文字溢出或留白）。

**建议修正**（二选一）：
- **方案 A**（推荐）：spec §3.1 给 `NumberBadgeStyle` 加 `fontFamily?: string` 字段（跟主标注一致），spec §6.1 回调签名改为 `(text, fontSize, fontFamily) => number`，plan Task 3 Step 1 显式要求复用/抽提 `measureTextWidth`。
- **方案 B**：spec §6.1 注明 badge 固定用 `system-ui` 字族测量（不跟随主标注 font），plan Task 3 Step 1 写死字族。

### 🟡 2.3【建议】plan Task 4 "Possibly use" 措辞让 subagent 难判断

**plan 原文**（Task 4 Files）：
```
- Possibly use: `src/canvas/labelMetrics.ts`, `src/geometry/labelBox.ts`, existing coordinate helpers
```

**问题**：「Possibly use」是给人类读者的弹性措辞，但 plan 的读者是 subagent（AGENTS.md 明确要求每个 subagent 只读对应 Task）。subagent 遇到"Possibly"会困惑到底用不用，可能跳过去自己实现一份，造成重复代码。

**实际依赖关系**（已核查）：
- `labelMetrics.ts` —— **必须用**，因为 Task 4 Step 5 要求 smart badge 的 label box 跟主标签用同一套测量，否则 badge 位置跟标签对不齐
- `labelBox.ts` —— **必须用**，smart badge 的 label anchor 要复用主标签的位置计算

**建议修正**：把 plan Task 4 Files 的"Possibly use"改为明确依赖：
```
- Required reuse: `src/canvas/labelMetrics.ts` (label measurement), `src/geometry/labelBox.ts` (smart label anchor)
```

## 3. 澄清项（4 处，spec/plan 未说清，需执行时判断）

这些不是错误，是 spec/plan 留白，建议执行前明确。

### 🟡 3.1 `addAnnotation` 签名两种写法未拍板

**spec §4.2**：
```typescript
addAnnotation: (a: Annotation, options?: { consumeNumber?: boolean }) => void;
// 或
addNumberedAnnotation: (a: Annotation) => void;
```

**plan Task 2 Step 3** 倾向第一种 `{ consumedNumber?: boolean }`（注意 plan 用 `consumedNumber`，spec 用 `consumeNumber`，**命名不一致**）。

**建议**：执行时统一用 plan 的写法（`consumedNumber`，语义更准 —— "已消费的编号"），并在 Task 2 commit 前对齐 spec 文档。

### 🟡 3.2 smart badge 的 label box 在无 note 时如何处理

**spec §6.7**：
> - 缺少 `rect` 或 `arrow`,回退到 label box 左侧;再缺失则不渲染 badge。

**问题**：智能标注在用户**还没输入文字**时（绘制中或空 note），`note === undefined`，此时 `TextLabelShape.tsx:27` 会 return null（不渲染标签）。如果此时 anchor=label，badge 就没有 label box 可依附。

**影响**：用户画完智能标注框 + 箭头，但还没敲文字回车时，badge 不显示。等文字输入完才出现。这可能是期望行为（badge 跟标签一起出现），也可能不是（用户希望 badge 在箭头定终点时就出现）。

**建议**：plan Task 4 Step 5 明确：「smart badge 在 annotation 提交到 store 后渲染。若 anchor=label 且 note 为空，badge 暂不渲染；note 填入后自动出现」。或反过来：「badge 在标注提交时立即渲染，anchor=label 时用箭头 endX/endY 作为 label box 回退位置」。**需要用户/产品判断**。

### 🟡 3.3 plan Task 5 Step 4 text 工具的 commit 时机

**plan 原文**（Task 5 Step 4）：
> If text annotation creation is delayed until text input confirms, consume the number on final commit, not when opening the input overlay.

**问题**：措辞是条件句"If... delayed"，但代码实际就是延迟提交的（`useTextTool.ts` 通过 `TextInputOverlay` 收输入，回车才入库 —— 已从 `EditorStage.tsx:32-33` 的 text handlers 结构推断）。plan 用条件句会让 subagent 以为存在"非延迟"路径。

**建议**：plan Task 5 Step 4 改为肯定句：「Text annotation 在用户输入并回车确认后才入库。`applyNumberBadgeIfEnabled` 必须在最终入库时调用，不能在打开 input overlay 时调用 —— 否则用户 ESC 取消会白消耗一个编号。」

### 🟡 3.4 plan Task 6 缺少 mosaic 工具的明确排除说明

**现状**：plan Task 6 的 UI 控件只覆盖 smart/rect/arrow/text 四个工具面板（正确），但**没显式说**「mosaic 工具面板不加编号控件」。

**风险**：如果 subagent 在 Task 6 看到 StylePanel 里有 `tool === "mosaic"` 的分支（虽然现状没有），可能顺手加上 mosaic 编号控件，违反 PRD 排除项。

**建议**：plan Task 6 Implementation Notes 加一条：「Do not add numbering controls to mosaic panel. Mosaic is excluded from numbering per PRD.」
